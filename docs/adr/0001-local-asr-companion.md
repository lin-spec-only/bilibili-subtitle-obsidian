# ADR-0001：使用本地 ASR 伴随服务

## 状态

已接受

## 背景

扩展目前只能读取 B 站已有字幕。无字幕视频需要下载音轨并运行语音识别，但 Manifest V3 后台无法可靠执行 FFmpeg 或大型模型。用户要求隐私优先，并将模型、环境和缓存放在空间充足的 D 盘。

## 决策

- 保留“官方字幕优先”的现有路径。
- 无可用字幕时，由扩展读取当前分 P 的短期音频地址，并把地址发送给仅监听 `127.0.0.1:8766` 的 Python 服务；8765 已被本机现有服务占用。
- 本地服务使用 faster-whisper，默认 `small`、CPU `int8`；模型和缓存目录由启动脚本固定到 D 盘。
- 服务只接受 HTTPS 的 B 站音频 CDN 地址，不接收 Cookie、token 或任意本地文件路径。
- 转写结果转换为现有 `{from, to, content}` 结构，继续复用 Markdown、SRT 和 Obsidian 导出。

## 后果

### 正面

- 音频和字幕不上传第三方。
- 不破坏现有字幕提取与导出代码。
- 模型可在后续切换为 FunASR 或 GPU，而无需改导出层。

### 负面

- 使用前需要启动本地服务。
- CPU 首次加载模型和长视频转写需要等待。
- 未升级显卡驱动前不默认启用 CUDA。

### 中性

- 单用户场景一次只执行一个转写任务；结果按视频和分 P 缓存在本地。

## 考虑过的替代方案

- 浏览器内 WebAssembly/WebGPU：模型下载和内存压力较大，MV3 生命周期不适合长任务。
- 云端 ASR：接入简单，但会上传音频并产生持续费用，与当前隐私承诺不一致。
- Native Messaging：安全边界更强，但首次注册和扩展 ID 配置复杂，适合作为后续产品化升级。
